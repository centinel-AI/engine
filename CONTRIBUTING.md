# Contributing to Grauss

Thank you for your interest in contributing. This document explains how to propose changes, what conventions to follow, and what to expect from the review process.

## Clouds and providers

Azure in this repo means **AzureRM + AzureAD** in the same `providers/azure/` tree (resources and Entra / directory). **OCI** uses `providers/oci/` and `oracle/oci` ([registry](https://registry.terraform.io/providers/oracle/oci/latest)) — see [`.cursor/rules/grauss-providers-and-clouds.mdc`](.cursor/rules/grauss-providers-and-clouds.mdc) for links and Compose auth env.

## Getting started

1. Fork the repository and clone your fork.
2. Copy `.env.example` to `.env` (`task env:init`) and set your credentials. For plan/apply against real projects, clone [centinel-AI/data](https://github.com/centinel-AI/data) and set `ENGINE_DATA_REPO_PATH`.
3. Run `task generate:<cloud>` to populate `providers/<cloud>/`.
4. Verify changes with `terraform fmt -check` and `terraform validate` before opening a pull request.

## Reporting bugs

Open a GitHub issue and include:

- A clear description of the unexpected behavior.
- The Terraform / OpenTofu version (`terraform version` or `tofu version`).
- The provider version in use (from `.env.example` or `.env`).
- The minimal JSON input path in the [data repository](https://github.com/centinel-AI/data) that reproduces the problem.
- The full error output (redact real credentials).

## Proposing changes

For non-trivial changes, open an issue first to discuss the approach before writing code. This avoids wasted effort when a proposal conflicts with the project direction.

For small fixes (typos, formatting, documentation), open a pull request directly.

## Development conventions

### Repository layout

```
providers/<cloud>/          # generated .tf templates (gitignored except .gitkeep)
  _locals.tf                # shared variable and json_data loader
  _<cloud>.tf.json          # required_providers + provider config (version placeholders)
  <resource_type>.tf        # one file per resource type, generated by scripts/codegen.py

workspace/                  # host mirror of /app/workspace (gitignored)
scripts/                    # codegen, version injection, workspace population
Dockerfile                  # engine image (CLOUD_PROVIDER + ENGINE build args)
compose.yml                 # ten engine services
```

Project JSON lives in the separate [data repository](https://github.com/centinel-AI/data); the engine reads it at run time via `ENGINE_DATA_REPO_PATH`.

### Template rules

- **No Terraform modules.** Use native constructs only (`for_each`, `try()`, `for` expressions, `jsondecode`).
- `try()` for optional JSON fields; never `lookup()` or `optional()`.
- Resource label is always `grauss`.
- All identifiers use `snake_case`.
- Each `.tf` file contains exactly one resource block plus the `for_each` and argument assignments.
- No hardcoded values in `.tf` files — all configuration comes from external JSON at run time.

### Data rules

JSON conventions are defined in the [data repository](https://github.com/centinel-AI/data). Engine changes should not add or track project JSON under this repo.

### Formatting and validation

```bash
terraform fmt -recursive providers/
cd workspace && terraform validate     # after task workspace:<cloud>
```

The CI pipeline enforces `terraform fmt -check`.

## Pull request process

1. Branch from `main` using the pattern `feat/<resource>`, `fix/<description>`, or `docs/<description>`.
2. Keep pull requests focused on a single resource type or concern.
3. Update the relevant rule(s) under `.cursor/rules/` if you introduce or change a convention.
4. Ensure `terraform fmt -check` and `terraform validate` pass.
5. A maintainer will review within a reasonable timeframe. Address all comments before the PR is merged.
